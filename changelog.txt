v1.0.20 — 2026-06-12
---------------------
New Feature: Chunk Claim — claim an entire chunk with a single GUI interaction.

OVERVIEW:
Players can now claim a full 16×16 chunk column from a single GUI menu.
No text input, no commands — the entire flow is GUI-driven and commandless,
consistent with the server's existing design philosophy.

CHANGED — New flow:
  Sneak + right-click an unclaimed chest with shears now opens a
  "Choose Claim Type" GUI instead of immediately starting a custom session.

  Slot 11 — Custom Claim (GRASS_BLOCK): starts the existing pos1/pos2 workflow.
  Slot 15 — Chunk Claim (BLUE_CONCRETE): auto-claims the 16×16 chunk containing
             the anchor chest; shows a green particle preview of exact chunk
             boundaries; opens a confirm GUI showing cost and upkeep before charging.

  After clicking "Chunk Claim":
    - Chunk bounds are computed from the anchor chest's chunk (chunkX*16 to
      chunkX*16+15, full world height, chunkZ*16 to chunkZ*16+15).
    - Green particle outline (0,255,128) renders the exact chunk perimeter.
    - A hologram above the chest shows "Chunk Claim / Your chunk is highlighted /
      Right-click the chest to confirm".
    - Confirm GUI shows: Cost: 15 Bops, Upkeep: 2 Bops/day.
    - Player confirms; Bops are only charged on confirmation.
    - Insufficient Bops shows the existing "insufficient-funds" message.
    - Overlap with existing claim shows the existing "claim-overlap" message.

CHANGED — Protection and upkeep:
  Chunk claims use the same ACTIVE/INACTIVE state machine as custom claims.
  dailyUpkeepCost is stored as a flat value (from chunk-claims.upkeep-bops-per-day),
  so the existing midnight upkeep cycle handles chunk claims automatically.
  INACTIVE chunk claims lose protection exactly as custom claims do.

CHANGED — config.yml additions:

v1.0.19 — 2026-06-12
---------------------
Economy Rebalance: claim purchase costs and upkeep significantly reduced.

CONTEXT:
Real server data showed the economy was unworkable — the wealthiest player had
~140 Bits, most active players had 20–80 Bits, and Bits are intentionally scarce.
At 1.0 Bop/block, even a tiny 10×5×10 claim (500 blocks) cost 500 Bops, which
was more than most players owned outright. Upkeep at 3% compounded the problem.

CHANGED — config.yml:
- claims.price-per-block: 1.0 → 0.02
  A 500-block claim (10×5×10) now costs 10 Bops (was 500).
  A 1,000-block claim now costs 20 Bops (was 1,000).
  A 2,000-block claim now costs 40 Bops (was 2,000).
  A 5,000-block claim now costs 100 Bops (was 5,000).

- upkeep.daily-cost-percent-of-purchase: 3.0 → 2.0
  At the new purchase prices, upkeep absolute costs also drop proportionally.
  A 1,000-block claim (20 Bops purchase) costs 0.40 Bops/day (was 30 Bops/day).
  A 2,000-block claim (40 Bops purchase) costs 0.80 Bops/day (was 60 Bops/day).
  A 5,000-block claim (100 Bops purchase) costs 2.00 Bops/day (was 150 Bops/day).

UNCHANGED:
- max-volume: 50,000 blocks
- All progression and scaling logic (volume × price, percent upkeep)
- All GUI layouts, protection logic, and upkeep mechanics
- All display code reads stored dailyUpkeepCost — newly created claims
  automatically reflect the new rates; existing claims are unaffected.

CHANGED — Java fallback defaults updated to match new config values in:
  ClaimConfirmGUI.java    — price-per-block 1.0→0.02, daily-cost-percent 3.0→2.0
  GUIListener.java        — price-per-block 1.0→0.02, daily-cost-percent 3.0→2.0
  ClaimSetupListener.java — price-per-block 1.0→0.02
  ClaimStorage.java       — daily-cost-percent 3.0→2.0 (migration fallback)

v1.0.16 — 2026-06-10
---------------------
Automatic Upkeep System (Rust-style midnight auto-pay).

OVERVIEW:
Claims now maintain themselves automatically.
Bops are withdrawn from the owner's balance at midnight each day.
No Claim Bank, no Claim Shop, no manual payment — fully automatic.

NEW — Midnight upkeep cycle:
- Fires once per day at midnight (server wall-clock time).
- On startup, schedules the cycle for the next midnight automatically.
- If upkeep.enabled = false, the cycle is a no-op.

NEW — Per-owner most-expensive-first processing:
- All ACTIVE claims for an owner are sorted most-expensive-first.
- Claims are greedily marked affordable (balance is tracked as claims are assigned).
- All affordable claims are charged in a single Bops withdrawal.
- Expensive claims fail before cheap ones when funds are limited.
  Example: balance=200, claims C=300 B=150 A=50.
    C → cannot afford (300 > 200), fails.
    B → can afford (150 ≤ 200), paid. running balance = 50.
    A → can afford (50 ≤ 50), paid.
    Total withdrawn: 200 Bops. C deactivated.

NEW — Online payment failure (5-minute countdown):
- If owner is ONLINE and a claim cannot be paid at midnight:
    1. Claim is marked "Payment Due" (runtime only, not persisted).
    2. Owner receives chat message: "Your claim requires N Bops for upkeep.
       You have 5 minutes to obtain enough Bops or protection will end."
    3. Server checks every 5 seconds if the owner now has enough Bops.
    4. If yes: deducts automatically, claim stays ACTIVE, confirms in chat.
    5. If owner goes OFFLINE during countdown: claim immediately INACTIVE.
    6. If 5 minutes expire without payment: claim becomes INACTIVE.

NEW — Offline payment failure (instant):
- If owner is OFFLINE and a claim cannot be paid: claim immediately INACTIVE.
  No countdown, no warning, no delay.

NEW — Hologram "Payment Due" state:
- While a claim is in the online countdown, hologram shows "Payment Due" (yellow).
- Returns to "Active" (green) if payment succeeds.
- Becomes "Inactive" (red) if countdown expires or owner goes offline.

NEW — First day free:
- Claims created today (after the most recent midnight) are skipped by the
  current midnight cycle. First upkeep charge is at the NEXT midnight.
  Example: created Monday 4 PM → first charge Tuesday midnight.

NEW — /claims cycle (admin command):
- Manually triggers the midnight upkeep cycle for testing.
- Requires chestclaims.admin permission (same as /claims reload).

NEW — Claim deletion cancels countdown:
- If a claim is deleted while in a Payment Due countdown, the countdown
  task is cancelled immediately to prevent any further state changes.

REMOVED — Periodic upkeep state check:
- The old upkeep.check-interval-minutes config key and periodic checkUpkeep()
  task are removed. The midnight cycle replaces this entirely.
- Old manual payUpkeep() / payUpkeepMax() methods removed from UpkeepManager
  (Claim Shop was already removed in v1.0.14).

CHANGED — config.yml:
- Removed: upkeep.check-interval-minutes (no longer used)

CHANGED — plugin.yml:
- Updated /claims usage: added "cycle" subcommand.

CHANGED — Files modified:
  UpkeepManager.java        — complete rewrite: midnight cycle, auto-pay, countdown
  ChestClaimsPlugin.java    — removed periodic checkUpkeep(); added upkeepManager.init()/shutdown()
  ClaimHologramManager.java — buildUpkeepStatusLine() now shows "Payment Due" (yellow)
  GUIListener.java          — deleteClaim() cancels countdown before removing claim
  ClaimsCommand.java        — added /claims cycle subcommand
  config.yml                — removed check-interval-minutes
  plugin.yml                — updated /claims usage string

v1.0.6 — 2026-06-09
---------------------
Upkeep Foundation.

NEW — Claim states:
- ACTIVE   : claim is protected and upkeep is current (or upkeep not yet configured).
- GRACE    : upkeep has expired; claim is still protected during the grace window.
- INACTIVE : grace period has elapsed; claim loses all protection.
- Transitions: ACTIVE → GRACE when upkeep-paid-until passes.
               GRACE → INACTIVE when grace-period-hours after grace start elapses.
               Any state → ACTIVE when the player pays upkeep (upkeep-paid-until extended).

- state                  : ACTIVE / GRACE / INACTIVE (default ACTIVE on migration)
- upkeep-paid-until      : epoch ms of when upkeep expires (0 = not yet set)
- grace-started-at       : epoch ms of when grace period began (0 = not in grace)
- daily-upkeep-cost      : precomputed daily cost in Bops (cost × daily-cost-percent / 100)

NEW — Migration:
- Existing claims load safely: state defaults to ACTIVE, upkeep-paid-until defaults to 0.
- Claims with upkeep-paid-until = 0 are skipped by the state-check task (never expire).
- daily-upkeep-cost recomputed from config on first load if missing from the file.
