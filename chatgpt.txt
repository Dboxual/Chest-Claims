LATEST SESSION — 2026-06-16 (v1.0.24)
--------------------------------------

Version bumped: 1.0.23 → 1.0.24

BUG FIXED
  The Chunk Selector (NETHER_STAR given during the Add Chunk flow) could be used
  as a normal item: placed in crafting grids to produce recipes (e.g. a beacon),
  moved into chests/barrels/furnaces/hoppers, or transported by automated systems.

ROOT CAUSE
  ChunkSelectorListener only protected the item from being dropped (PlayerDropItemEvent)
  and removed it on logout (PlayerQuitEvent). No protection existed against inventory
  interactions outside the plugin's own chunk-selection flow.

FIX SUMMARY
  All ChestClaims plugin-utility items now carry a second PDC tag:
    key:  chestclaims:plugin-item  (namespace: chestclaims, key: plugin-item)
    type: BYTE, value: 1
  This tag is applied by ChunkSelectorListener.buildSelectorItem() at item-creation time.
  A new listener, PluginItemProtectionListener, blocks all non-intended use by checking
  this tag rather than display names.

NEW FILE — PluginItemProtectionListener.java
  Handles four events:

  1. InventoryClickEvent (HIGH priority, ignoreCancelled = false)
     Detects when the cursor item OR the slot's current item is a plugin item.
     • Top inventory type == PLAYER → allowed (player freely rearranges own inv).
     • Top inventory type == CRAFTING (2×2 survival screen):
         – Click lands in the crafting grid (top inv) → cancelled.
         – Shift-click (would send item to crafting area)  → cancelled.
         – Other clicks within player inv area             → allowed.
     • All other top inventory types (CHEST, BARREL, FURNACE, BLAST_FURNACE,
       SMOKER, HOPPER, DROPPER, DISPENSER, BREWING, WORKBENCH, ANVIL, GRINDSTONE,
       SMITHING, LOOM, CARTOGRAPHY, STONECUTTER, ENDER_CHEST, SHULKER_BOX, BEACON,
       ENCHANTING, and any future type) → event cancelled unconditionally.

  2. InventoryDragEvent (HIGH priority, ignoreCancelled = false)
     If the old cursor (dragged item) is a plugin item and any inventory other than
     PLAYER is open → cancelled. Prevents dragging the item into any non-player slot.

  3. PrepareItemCraftEvent (HIGH priority)
     Safety net: if any ingredient in the crafting matrix is a plugin item, the
     crafting result is cleared (set to null). Fires for both the 2×2 survival grid
     and 3×3 crafting tables. Ensures no recipe output is possible even if the item
     somehow ends up in the grid through an unconventional path.

  4. InventoryMoveItemEvent (HIGH priority, ignoreCancelled = true)
     Cancelled whenever the moved item is a plugin item. Covers automated transport
     via hoppers, dispensers, and droppers, including cross-container suction.

CHANGE — ChunkSelectorListener.buildSelectorItem()
  Added pluginItemKey (chestclaims:plugin-item) field, initialized in constructor.
  When the Chunk Selector item is created, it now receives two PDC tags:
    chestclaims:chunk-selector = <claimUUID string>  (existing — identifies the claim)
    chestclaims:plugin-item    = (byte) 1             (NEW — marks it as a protected item)

CHANGE — ChestClaimsPlugin.onEnable()
  Instantiates PluginItemProtectionListener and registers it as an event listener.
  Import added: com.chestclaims.listener.PluginItemProtectionListener

IDENTIFICATION RULE (enforced by code)
  Plugin utility items MUST be identified by PersistentDataContainer tags, never by
  display names. The chestclaims:plugin-item byte tag is the canonical marker.
  Any future plugin-given item must set this tag in its builder method.

LATEST SESSION — 2026-06-15 (v1.0.23)
--------------------------------------

Version bumped: 1.0.22 → 1.0.23

ROOT CAUSE ADDRESSED
  Previously, ClaimProtectionListener only protected against player block-break/place events.
  All environmental grief vectors (explosions into claims, fire, liquid flow, container looting)
  were unprotected. ACTIVE claims now have comprehensive environmental protection.

CHANGE 1 — Explosion protection extended to all claim blocks (EntityExplode, BlockExplode)

  WHAT WAS BROKEN
    TNT placed outside a claim could destroy blocks inside it. Creeper, wither, ender
    crystal, TNT minecart, and bed/respawn-anchor explosions all bypassed claim protection.
    Previously, only the anchor block (claim chest) was removed from explosion block lists.

  FIX
    `isExplosionProtected(Location)` now returns true if:
      - The location is a claim anchor (always protected, any state), OR
      - The location is inside any ACTIVE claim.
    Both `onEntityExplode` and `onBlockExplode` use this helper to strip protected blocks
    from the explosion's block list before destruction occurs.

CHANGE 2 — Container access protection (PlayerInteractEvent)

  WHAT WAS BROKEN
    Non-authorized players could open and loot any inventory-holding block (furnace, hopper,
    shulker box, brewing stand, etc.) inside someone else's ACTIVE claim simply by right-clicking.
    Only the anchor block (claim chest) had access protection via ClaimInteractListener.

  FIX — onContainerInteract
    When a player right-clicks any block whose BlockState implements
    org.bukkit.block.Container (i.e., any inventory-holding block), inside an ACTIVE claim:
      - Owner, trusted players, team members (when enabled), admins → allowed.
      - Everyone else → event cancelled, action-bar: "You can't open containers in this claim."
    Anchor blocks are explicitly skipped here (ClaimInteractListener owns their handling).

  COVERED BLOCK TYPES (via org.bukkit.block.Container interface):
    Chest, Trapped Chest, Barrel, Shulker Box (all 16 colors), Furnace, Blast Furnace,
    Smoker, Hopper, Dropper, Dispenser, Brewing Stand, Crafter, Lectern, Chiseled Bookshelf,
    and any future block type that implements the Container interface.

CHANGE 3 — Bucket-fill protection (PlayerBucketFillEvent)

  WHAT WAS BROKEN
    A player could fill a bucket from a water or lava source inside a claim they don't own,
    effectively stealing liquid without triggering any block-break protection.

  FIX — onBucketFill
    Cancelled when the liquid source block is inside an ACTIVE claim and the player lacks access.
    Message: "You can't take from blocks inside this claim."
    Note: Bucket-empty (placing liquid) is already covered by the existing BlockPlaceEvent handler.

CHANGE 4 — Fire burn protection extended to all claim blocks (BlockBurnEvent)

  WHAT WAS BROKEN
    Only the anchor block was protected from burning. All other blocks inside an ACTIVE claim
    could be destroyed by fire.

  FIX — onBlockBurn
    Cancelled for any block inside an ACTIVE claim (in addition to anchor blocks, which remain
    always protected regardless of claim state).

CHANGE 5 — Fire spread prevention (BlockSpreadEvent, BlockIgniteEvent)

  WHAT WAS BROKEN
    Fire outside a claim could spread into it, creating fire blocks inside and eventually
    burning claim contents (with BlockBurnEvent now cancelled, fire blocks would sit forever
    and look griefed). Additionally, players with flint & steel or fire charges could ignite
    blocks inside someone else's claim.

  FIX — onBlockSpread
    Cancelled when the new block state would be FIRE or SOUL_FIRE and the target location is
    inside an ACTIVE claim. Prevents fire from entering the claim boundary.

  FIX — onBlockIgnite
    Two cases cancelled inside ACTIVE claims:
      1. Player-caused (flint & steel, fire charge): cancelled if player lacks access.
         Message: "You cannot start fires inside this claim."
      2. Lava-caused (IgniteCause.LAVA): cancelled unconditionally to prevent lava adjacent
         to a claim from igniting blocks inside it.
      Lightning: intentionally NOT cancelled (acts of God).
      Fire spreading (IgniteCause.SPREAD): handled by BlockSpreadEvent above.

CHANGE 6 — Liquid grief prevention (BlockFromToEvent)

  WHAT WAS BROKEN
    Water or lava placed outside an ACTIVE claim could flow into it, flooding or griefing
    the interior. Lava could additionally destroy items and blocks inside the claim.

  FIX — onBlockFromTo
    Cancelled when:
      - The source block is liquid (water or lava), AND
      - The destination block is inside an ACTIVE claim, AND
      - The source block is NOT inside the same claim.
    This allows owner-placed liquid to flow freely within their own claim boundaries, while
    external liquid is stopped at the claim border. Liquid from claim A is also blocked from
    flowing into claim B.

KNOWN REMAINING GAPS (documented, not fixed in this pass)
----------------------------------------------------------
These grief vectors are NOT yet protected. Add in a future pass if needed:

1. Piston extension/retraction
   A piston outside a claim could push blocks into or pull blocks from an ACTIVE claim.
   Fix: BlockPistonExtendEvent + BlockPistonRetractEvent — check pushed block destinations.

2. Endermen stealing blocks
   Endermen can pick up certain blocks (dirt, gravel, concrete powder, etc.) inside claims.
   Fix: EntityChangeBlockEvent where entity is Enderman + block is inside ACTIVE claim.

3. Hopper pulling items through walls
   A hopper placed outside but immediately below an ACTIVE claim's container border
   may be able to pull items out of a container one block above inside the claim.
   Fix: InventoryMoveItemEvent — check if destination inventory is outside the claim of
   the source inventory.

4. Silverfish / zombie griefing
   Silverfish infect stone blocks; zombies break doors. Both trigger EntityChangeBlockEvent.
   Fix: EntityChangeBlockEvent for the relevant entity types inside ACTIVE claims.

5. Wither spawning destruction
   The wither's initial spawn explosion is an EntityExplodeEvent (now covered), but the
   wither's blue skull bypasses blast resistance. Covered by EntityExplodeEvent.
   Wither skull direct hits on blocks: EntityChangeBlockEvent. Likely low priority.

Version bumped: 1.0.21 → 1.0.22

CHANGE 1 — Claim chest (anchor) protection hardened

  WHAT WAS BROKEN
    - Non-owners could break claim anchor blocks if the claim was INACTIVE (no upkeep).
    - Explosions (TNT, creeper, bed/respawn anchor) could destroy the anchor block, leaving
      a ghost claim in storage with no physical chest and no way to clean it up in-game.
    - Fire could spread to and burn barrel-type anchors.

  FIX — ClaimProtectionListener.java
    onBlockBreak: Removed the INACTIVE bypass for anchor blocks. The anchor is now ALWAYS
    protected from non-owner breaking, regardless of claim state (ACTIVE or INACTIVE).
      - Owner and admins (chestclaims.admin) → delete GUI.
      - Anyone else → event cancelled, action-bar: "You can't break this claim chest."
      - Message unified to a single clear line instead of two different messages.
      - Admin bypass: ops with chestclaims.admin permission can open the delete GUI on any claim.

    onEntityExplode (NEW): removes anchor blocks from explosion block lists so TNT and
    creepers cannot destroy claim chests.

    onBlockExplode (NEW): same protection against block-source explosions (beds in Nether,
    respawn anchors in Overworld, etc.).

    onBlockBurn (NEW): cancels fire-burn events targeting anchor blocks (protects barrel
    anchors from fire spread).

CHANGE 2 — Ghost/floating outline prevention

  WHAT WAS BROKEN
    If a claim anchor block was somehow removed (server crash, world edit, pre-fix explosion)
    without going through the proper deletion flow, the claim remained in storage with
    outlineEnabled = true, causing particles to keep rendering in the world with no physical
    chest underneath — a "ghost outline."

  FIX — ClaimOutlineTask.java (runtime check)
    Before rendering any outline for a claim, ClaimOutlineTask now verifies that the anchor
    block at claim.getAnchor() is still a CHEST, TRAPPED_CHEST, or BARREL in a loaded chunk.
    If the block is gone or wrong material:
      - Claim is removed from ClaimStorage.
      - Hologram is removed via ClaimHologramManager.onClaimDeleted().
      - Claims file is saved immediately.
      - Warning logged: "Claim <id> owned by <name> has no anchor block at X,Y,Z in '<world>' — removing orphaned claim."
    Orphans are collected during the iteration, cleaned up after, so no ConcurrentModification.

  FIX — ClaimStorage.removeClaimsWithMissingAnchors() (startup scan)
    New method scans all loaded claims at plugin startup. For each claim whose anchor block
    is not a chest/barrel:
      - Removes it from the claims list.
      - Saves the updated claims.yml.
      - Returns the removed list so the caller can log each removal.
    Skips claims in worlds that are not currently loaded (no false positives for unloaded worlds).

  FIX — ChestClaimsPlugin.onEnable() (startup validation)
    After claimStorage.load() and before hologramManager.init(), calls
    claimStorage.removeClaimsWithMissingAnchors(). Each removed claim is logged at WARNING
    level with its ID, owner, coordinates, and world. Total count logged at INFO level.
    This means no hologram is ever spawned for a claim with a missing anchor.

BUGFIX SESSION — 2026-06-15 (chunk claim outline rendering)
------------------------------------------------------------

WHAT WAS BROKEN
  Chunk claim outlines were effectively invisible. The previous implementation called
  OutlineTask.drawOutline() — a 3D cuboid renderer — for chunk claims, passing pos1 and
  pos2 whose Y values span the full world height (minHeight = -64, maxHeight - 1 = 319,
  a 383-block range). The vertical edges had only 15 particles each (the maxPerEdge cap),
  making them 1 particle per ~25 blocks and invisible in practice. The horizontal cap edges
  were drawn at y = -64 (underground) and y = 320 (above the sky), neither of which are
  visible to a player standing at ground level.

  As a result: toggling the outline ON for a chunk claim did nothing visible in-game.

ROOT CAUSE
  OutlineTask is designed for finite 3D selections where pos1/pos2 form a compact box.
  Chunk claims store pos1/pos2 spanning the FULL WORLD HEIGHT (set by startChunkClaimSession
  to world.getMinHeight() / world.getMaxHeight()-1). Drawing a 3D cuboid on this range
  was always wrong for the chunk-claim use case.

FIX
  Replaced the per-chunk-cuboid approach with a horizontal perimeter ring. For each claimed
  chunk, up to 4 horizontal edges are drawn at the player's current Y + 1. An edge is only
  drawn if the neighboring chunk is NOT in the claim's claimedChunks set — so internal
  shared edges are skipped and only the outer boundary is drawn.

  A new package-private static method OutlineTask.spawnLine() provides the particle line
  primitive. Both ClaimOutlineTask and ClaimBorderTask call it.

HOW CHUNK OUTLINE RENDERING NOW WORKS
  ClaimOutlineTask (owner-toggled persistent outline, all nearby players):
    For each chunk claim with outlineEnabled = true:
      For each online player within claim-outline.view-distance-blocks of the anchor:
        → For each chunk key in claimedChunks:
             Check 4 orthogonal neighbors; skip edges whose neighbor is also claimed.
             Draw surviving edges as horizontal DUST particle lines at (player.Y + 1).
        → Uses claim.getOutlineColor() for the particle color.
        → Particle density: selection-outline.max-particles-per-edge (default 15) per
          16-block edge → ~1 particle/block, same density as custom claim edges.

  ClaimBorderTask (shears-preview, per-player while holding shears):
    Identical horizontal perimeter logic, color = selection-outline.particle-color (orange).

  Custom (non-chunk) claims are unchanged: still use 3D cuboid via OutlineTask.drawOutline().

  PARTICLE COUNT PER TICK (worst case, max-particles-per-edge = 15):
    1-chunk claim:  4 outer edges × 16 particles = 64 particles per player per tick
    2-chunk claim:  6 outer edges × 16 particles = 96 particles
    N-chunk claim:  at most (2N+2) outer edges    — well under any spam threshold

  CLEANUP / LIFECYCLE (unchanged from before):
    Particles are ephemeral (DUST, ~1.5 s lifetime). Toggling the outline OFF simply stops
    scheduling new particles; old ones expire naturally within 2 ticks. Player logout, claim
    removal, and claim-outline.enabled = false all stop the rendering on the next task tick.

KNOWN LIMITATION
  The view-distance check in ClaimOutlineTask uses distance from the anchor block.
  For large multi-chunk claims the far end may be >32 blocks from the anchor, putting
  part of the perimeter outside the default view distance. Increase
  claim-outline.view-distance-blocks in config.yml if needed.

WORLDGUARD COMPATIBILITY
-------------------------
ChestClaims never overrides a WorldGuard (or any other protection plugin) cancellation.

Implementation:
- All block and interaction handlers run at EventPriority.HIGHEST with ignoreCancelled = true.
- WorldGuard uses EventPriority.HIGH, so it always processes and cancels events BEFORE ChestClaims.
- If an event is cancelled by the time ChestClaims runs, ChestClaims skips it entirely.

Affected handlers and what this means in practice:
  ClaimSetupListener.onInteract (HIGHEST, ignoreCancelled=true):
    → WG blocks interaction in a protected region → claim setup does not start, pos1/pos2
      not set, confirm GUI not opened.

  ClaimInteractListener.onChestInteract (HIGHEST, ignoreCancelled=true):
    → WG blocks interaction with a claimed chest → manage GUI does not open, even for owner.

  ClaimProtectionListener.onBlockBreak (HIGHEST, ignoreCancelled=true):
    → WG blocks a break → CC skips entirely. Owner does NOT get the delete GUI.
      Non-owner is already blocked by WG; CC adds nothing.

  ClaimProtectionListener.onBlockPlace (HIGHEST, ignoreCancelled=true):
    → WG blocks a place → CC skips entirely.

Key behavior: WorldGuard always wins. ChestClaims only acts on events that WorldGuard
(and other HIGH-priority plugins) have already allowed to proceed.

No WorldGuard API integration is required — this works purely via event priority ordering.


CLAIM OUTLINE SETTINGS SYSTEM (v1.0.21 + 2026-06-15 bugfix)
--------------------------------------------------------------
Owner right-clicks anchor → Claim Management GUI → slot 11 (Outline Settings) →
opens OutlineSettingsGUI (27 slots, see layout above).

OVERVIEW
--------
ChestClaims is a Paper 1.21.1 Minecraft plugin for creating and managing
land claims anchored to chests. Players use Shears as a FAWE-style 3D
selection wand to define a cuboid claim area. Claims cost Bops (currency
from the companion Bops plugin) per block of total volume, including air.

Claims have an upkeep system: a daily cost (configurable as a % of purchase
price per upkeep cycle, default 3%). Upkeep is processed automatically at
midnight — Bops are withdrawn from the owner's balance without any manual action.
If payment fails, online owners get a 5-minute countdown; offline owners lose
protection immediately. INACTIVE claims have no block protection but are not
auto-deleted.

Access management (v1.0.18): owners can trust currently online players through
the Select Player GUI and toggle BetterTeams teammate access per-claim from the
Claim Access GUI.

CLAIM STATES
------------
ACTIVE   — Upkeep is current. Full block protection.
INACTIVE — Upkeep unpaid and not resolved. No block protection.

Lifecycle: ACTIVE → INACTIVE via midnight cycle when payment fails.
Payment Due is a runtime-only countdown state (not persisted). ClaimState remains ACTIVE during
countdown; becomes INACTIVE only if countdown expires or owner goes offline.
GRACE was removed in v1.0.13. Old GRACE entries in claims.yml load as INACTIVE.


AUTOMATIC UPKEEP SYSTEM (v1.0.16)
-----------------------------------
Rust-style: claims auto-maintain by withdrawing from the owner's Bops balance at midnight.
No Claim Bank, no Claim Shop, no manual payment. Fully automatic.

MIDNIGHT CYCLE:
  - Fires once per day at midnight (server wall-clock time).
  - On plugin enable: next midnight is calculated and scheduled via runTaskLater.
  - After the first fire: repeating 24h runTaskTimer keeps subsequent cycles going.
  - /claims cycle (admin) manually triggers the cycle for testing.

PROCESSING ORDER (most expensive first):
  For each owner, claims are sorted by dailyUpkeepCost descending.
  Greedy assignment: if running balance ≥ claim cost → mark as affordable, deduct from running balance.
  Otherwise → mark as cannot-pay.
  All affordable claims are charged in one Bops withdrawal.
  Example: balance=200, claims sorted C=300, B=150, A=50.
    C: 300 > 200 → cannot pay.
    B: 150 ≤ 200 → pay. running balance = 50.
    A:  50 ≤  50 → pay. running balance = 0.
    Withdrawal: 200 Bops. C → fails.

ONLINE PAYMENT FAILURE (5-minute countdown):
  If owner is ONLINE and claim cannot be paid:
    1. Claim added to paymentDueClaims set (runtime only).
    2. Hologram shows "Payment Due" (yellow).
    3. Chat message: "Your claim requires N Bops for upkeep.
                      You have 5 minutes to obtain enough Bops or protection will end."
    4. Server checks every 5 seconds (via BukkitRunnable) if owner now has enough.
       - Has enough  → charge, keep ACTIVE, confirm in chat, cancel task.
       - Goes offline → immediately INACTIVE, save, cancel task.
       - 5 min expire → INACTIVE, notify in chat, cancel task.

OFFLINE PAYMENT FAILURE (instant):
  If owner is OFFLINE at midnight and claim cannot be paid:
  → Claim immediately becomes INACTIVE. No countdown, no warning.

FIRST DAY FREE:
  Claims created on or after today's midnight are skipped by the current midnight cycle.
  First upkeep charge is at the NEXT midnight.
  Example: created Monday 4 PM → first charge Monday→Tuesday midnight.

ANTI-EXPLOIT:
  The economic deterrent is the purchase cost. Deleting and recreating a claim costs the full
  block purchase price, which over time far exceeds the upkeep that would be owed.
  Technical note: claims created after the most recent midnight are skipped in the current cycle;
  the earliest a re-created claim can be charged again is the following midnight.

upkeepPaidUntil field (claims.yml):
  - Updated on successful payment (set to next midnight epoch ms).
  - Serves as an audit log; not used to drive the cycle logic.
  - Migration: existing claims with upkeepPaidUntil=0 participate in the next midnight cycle normally.

CLAIM CREATION FLOW (commandless)
-----------------------------------
1. Player holds Shears in main hand.
2. Sneak + right-click an unclaimed Chest / Trapped Chest / Barrel.
   → Session created. Setup hologram + particle outline task start.
3. Left-click any block → Pos1.
4. Right-click any block (not anchor) → Pos2.
   → Action bar: "Blocks: {vol} | Price: {cost} Bops"
5. Right-click SAME ANCHOR CHEST with both positions set → Confirm GUI opens:
     Slot 10: LIME_CONCRETE "Confirm Purchase"
              Lore: Claim Price / Upkeep per cycle (if upkeep enabled) / Click to purchase
     Slot 13: CHEST "Claim Summary"
              Lore: Claim Price / Upkeep per cycle (if upkeep enabled)
     Slot 16: RED_CONCRETE "Cancel"
6. Confirm → validates, charges Bops, saves claim, spawns hologram.
7. Cancel → session cancelled.
   Sneak+F or /claims cancel also cancels at any time.

dailyUpkeepCost stored on claim at creation = cost × (daily-cost-percent-of-purchase / 100).

-------------------------------------

   → "Claim Management" GUI opens (3 rows, 27 slots).

CLAIM DELETION FLOW
--------------------
1. Owner BREAKS their claim anchor chest.
   → Block break cancelled; "Delete Claim?" GUI opens.
   → Slot 10: RED_CONCRETE "Yes, Delete Claim"
   → Slot 13: BARRIER "Are you sure?"
   → Slot 16: LIME_CONCRETE "No, Keep Claim"

3. "No" → GUI closes, claim intact.
- No Bops refund. Works in all states.
