ACCESS RULES (v1.0.18)
----------------------
Owner:
  - Can always build/break/interact inside the claim.
  - Can always open the Claim Management GUI and Claim Access GUI.
  - Can always delete the claim (break anchor → delete GUI).

Trusted players (ACTIVE claim only):
  - Can build, break, and interact inside the claim.
  - Can open the anchor chest as a regular container.
  - Cannot delete the claim (breaking anchor shows "cannot break this claim's anchor chest").
  - Cannot open Claim Management GUI or Claim Access GUI.
  - Cannot toggle outline, edit trust settings, or change team access.

Team members (BetterTeams installed, team-access-enabled = true, ACTIVE claim only):
  - Same build/break/interact access as trusted players.
  - Same restrictions as trusted players.

Other non-owners (ACTIVE claim):
  - Blocked from break/place. Action-bar message shown.
  - Blocked from opening the anchor chest; action-bar "This claim belongs to <owner>."

INACTIVE claims:
  - No protection for anyone. All block events proceed normally.
  - Non-owners can open the anchor chest.


BETTERTEAMS INTEGRATION (v1.0.17)
-----------------------------------
Soft dependency — no BetterTeams jar required at compile time.

If BetterTeams is installed:
  - TeamsHook.init() detects it at startup. Logs "BetterTeams integration active."
  - TeamsIntegration uses reflection: com.booksaw.betterTeams.Team.getTeam(OfflinePlayer)
    then checks Team.getMembers() (Map<UUID, TeamPlayer>) for the playerUuid.
  - Team Access toggle in Claim Access GUI shows LIME_DYE / GRAY_DYE.
  - Toggling saves team-access-enabled per-claim to claims.yml.

If BetterTeams is not installed:
  - TeamsHook.isAvailable() returns false.
  - Team Access toggle shows BARRIER "BetterTeams is not installed." — clicking does nothing.
  - areTeammates() always returns false — no team bypass in protection.
  - Trusted player system works entirely independently.
  - No errors thrown.

WORLDGUARD COMPATIBILITY
-------------------------
ChestClaims never overrides a WorldGuard (or any other protection plugin) cancellation.

Implementation:
- All block and interaction handlers run at EventPriority.HIGHEST with ignoreCancelled = true.
- WorldGuard uses EventPriority.HIGH, so it always processes and cancels events BEFORE ChestClaims.
- If an event is cancelled by the time ChestClaims runs, ChestClaims skips it entirely.

Affected handlers and what this means in practice:
  ClaimSetupListener.onInteract (HIGHEST, ignoreCancelled=true):
    → WG blocks interaction in a protected region → claim setup does not start, pos1/pos2
      not set, confirm GUI not opened.

  ClaimInteractListener.onChestInteract (HIGHEST, ignoreCancelled=true):
    → WG blocks interaction with a claimed chest → manage GUI does not open, even for owner.

  ClaimProtectionListener.onBlockBreak (HIGHEST, ignoreCancelled=true):
    → WG blocks a break → CC skips entirely. Owner does NOT get the delete GUI.
      Non-owner is already blocked by WG; CC adds nothing.

  ClaimProtectionListener.onBlockPlace (HIGHEST, ignoreCancelled=true):
    → WG blocks a place → CC skips entirely.

Key behavior: WorldGuard always wins. ChestClaims only acts on events that WorldGuard
(and other HIGH-priority plugins) have already allowed to proceed.

No WorldGuard API integration is required — this works purely via event priority ordering.


CLAIM OUTLINE TOGGLE SYSTEM
-----------------------------
Owner right-clicks anchor → Claim Management GUI → slot 11 (Outline Toggle).
Clicking toggles claim.outlineEnabled and saves claims.yml immediately.

When outlineEnabled = true:
  ClaimOutlineTask (server-wide BukkitRunnable, runs every claim-outline.update-ticks ticks)
  → for each claim with outlineEnabled:
    → for each online player within claim-outline.view-distance-blocks (horizontal from anchor):
      → draws DUST particle outline (12 edges + 8 corner markers) around the claim
  Color: cyan (0,200,255) by default.
  Bedrock-compatible (DUST particles work via Geyser).

OVERVIEW
--------
ChestClaims is a Paper 1.21.1 Minecraft plugin for creating and managing
land claims anchored to chests. Players use Shears as a FAWE-style 3D
selection wand to define a cuboid claim area. Claims cost Bops (currency
from the companion Bops plugin) per block of total volume, including air.

Claims have an upkeep system: a daily cost (configurable as a % of purchase
price per upkeep cycle, default 3%). Upkeep is processed automatically at
midnight — Bops are withdrawn from the owner's balance without any manual action.
If payment fails, online owners get a 5-minute countdown; offline owners lose
protection immediately. INACTIVE claims have no block protection but are not
auto-deleted.

Access management (v1.0.18): owners can trust currently online players through
the Select Player GUI and toggle BetterTeams teammate access per-claim from the
Claim Access GUI.

CLAIM STATES
------------
ACTIVE   — Upkeep is current. Full block protection.
INACTIVE — Upkeep unpaid and not resolved. No block protection.

Lifecycle: ACTIVE → INACTIVE via midnight cycle when payment fails.
Payment Due is a runtime-only countdown state (not persisted). ClaimState remains ACTIVE during
countdown; becomes INACTIVE only if countdown expires or owner goes offline.
GRACE was removed in v1.0.13. Old GRACE entries in claims.yml load as INACTIVE.


AUTOMATIC UPKEEP SYSTEM (v1.0.16)
-----------------------------------
Rust-style: claims auto-maintain by withdrawing from the owner's Bops balance at midnight.
No Claim Bank, no Claim Shop, no manual payment. Fully automatic.

MIDNIGHT CYCLE:
  - Fires once per day at midnight (server wall-clock time).
  - On plugin enable: next midnight is calculated and scheduled via runTaskLater.
  - After the first fire: repeating 24h runTaskTimer keeps subsequent cycles going.
  - /claims cycle (admin) manually triggers the cycle for testing.

PROCESSING ORDER (most expensive first):
  For each owner, claims are sorted by dailyUpkeepCost descending.
  Greedy assignment: if running balance ≥ claim cost → mark as affordable, deduct from running balance.
  Otherwise → mark as cannot-pay.
  All affordable claims are charged in one Bops withdrawal.
  Example: balance=200, claims sorted C=300, B=150, A=50.
    C: 300 > 200 → cannot pay.
    B: 150 ≤ 200 → pay. running balance = 50.
    A:  50 ≤  50 → pay. running balance = 0.
    Withdrawal: 200 Bops. C → fails.

ONLINE PAYMENT FAILURE (5-minute countdown):
  If owner is ONLINE and claim cannot be paid:
    1. Claim added to paymentDueClaims set (runtime only).
    2. Hologram shows "Payment Due" (yellow).
    3. Chat message: "Your claim requires N Bops for upkeep.
                      You have 5 minutes to obtain enough Bops or protection will end."
    4. Server checks every 5 seconds (via BukkitRunnable) if owner now has enough.
       - Has enough  → charge, keep ACTIVE, confirm in chat, cancel task.
       - Goes offline → immediately INACTIVE, save, cancel task.
       - 5 min expire → INACTIVE, notify in chat, cancel task.

OFFLINE PAYMENT FAILURE (instant):
  If owner is OFFLINE at midnight and claim cannot be paid:
  → Claim immediately becomes INACTIVE. No countdown, no warning.

FIRST DAY FREE:
  Claims created on or after today's midnight are skipped by the current midnight cycle.
  First upkeep charge is at the NEXT midnight.
  Example: created Monday 4 PM → first charge Monday→Tuesday midnight.

ANTI-EXPLOIT:
  The economic deterrent is the purchase cost. Deleting and recreating a claim costs the full
  block purchase price, which over time far exceeds the upkeep that would be owed.
  Technical note: claims created after the most recent midnight are skipped in the current cycle;
  the earliest a re-created claim can be charged again is the following midnight.

upkeepPaidUntil field (claims.yml):
  - Updated on successful payment (set to next midnight epoch ms).
  - Serves as an audit log; not used to drive the cycle logic.
  - Migration: existing claims with upkeepPaidUntil=0 participate in the next midnight cycle normally.

CLAIM CREATION FLOW (commandless)
-----------------------------------
1. Player holds Shears in main hand.
2. Sneak + right-click an unclaimed Chest / Trapped Chest / Barrel.
   → Session created. Setup hologram + particle outline task start.
3. Left-click any block → Pos1.
4. Right-click any block (not anchor) → Pos2.
   → Action bar: "Blocks: {vol} | Price: {cost} Bops"
5. Right-click SAME ANCHOR CHEST with both positions set → Confirm GUI opens:
     Slot 10: LIME_CONCRETE "Confirm Purchase"
              Lore: Claim Price / Upkeep per cycle (if upkeep enabled) / Click to purchase
     Slot 13: CHEST "Claim Summary"
              Lore: Claim Price / Upkeep per cycle (if upkeep enabled)
     Slot 16: RED_CONCRETE "Cancel"
6. Confirm → validates, charges Bops, saves claim, spawns hologram.
7. Cancel → session cancelled.
   Sneak+F or /claims cancel also cancels at any time.

dailyUpkeepCost stored on claim at creation = cost × (daily-cost-percent-of-purchase / 100).


CLAIM MANAGEMENT FLOW (commandless)
-------------------------------------
1. Owner right-clicks their claim anchor chest.
   → "Claim Management" GUI opens (3 rows, 27 slots).
   → Slot 11: Outline toggle (LIME_DYE or GRAY_DYE)
   → Slot 13: CHEST info item (owner, price paid, upkeep cost per cycle)
2. Click Outline Toggle → toggles outlineEnabled, saves, GUI reopens.
3. Press Escape → GUI closes. No change.


CLAIM DELETION FLOW
--------------------
1. Owner BREAKS their claim anchor chest.
   → Block break cancelled; "Delete Claim?" GUI opens.
   → Slot 10: RED_CONCRETE "Yes, Delete Claim"
   → Slot 13: BARRIER "Are you sure?"
   → Slot 16: LIME_CONCRETE "No, Keep Claim"
2. "Yes" → claim removed, anchor breaks naturally (drops chest + contents).
3. "No" → GUI closes, claim intact.
- No Bops refund. Works in all states.


NON-OWNER INTERACTION
----------------------
- Non-owner right-clicks claimed chest → action bar: "This claim belongs to <owner>." No GUI.
- Non-owner breaks/places inside ACTIVE claim → blocked.
- Non-owner breaks/places inside INACTIVE claim → allowed.
